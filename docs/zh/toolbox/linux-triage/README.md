# linux-triage

> **输入：** 无 · **输出：** stdout 上一份分诊报告 · **风险：** 只读 ·
> **root：** 可选——日志错误和部分计数器在 root 下更全

每次事故的第一步：一条命令、一屏、全貌——负载 vs 核心数、CPU/内存占用前几名、
磁盘和 inode 压力、网卡地址、连接数、失败的 systemd 单元、最近 20 条日志错误。
任何值得警惕的项都会重复为一行 `FLAG:` 并计入退出码，这样人和自动化都能据此
判断"这里到底是不是真出问题了"。

## 用法

```bash
./linux-triage.sh            # 非特权：完整报告，journal 可能不全
sudo ./linux-triage.sh       # 全部
```

红旗阈值：load1 > 2× 核心数 · 磁盘或 inode 使用率 ≥ 90% · 任何失败的 systemd 单元。

## 退出码

| 码 | 含义 |
| --- | --- |
| 0 | 报告已生成，无红旗 |
| 1 | 发现红旗（数量见结论） |
| 2 | 非 Linux |

## 验证环境

Ubuntu 24.04 容器（非特权 + root）。使用 `/proc`、`ps`、`free`、`df`、`ip`、`ss`、
`systemctl`、`journalctl`——任何 systemd 发行版都标配；缺某个命令时优雅降级。
lab 验证过，非生产级加固。
